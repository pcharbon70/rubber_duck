# Feature: Repository Consolidation and Project-Based Data Organization

## Summary
Consolidate all separate repository modules (ConversationRepo, MessageRepo, EngineSessionRepo, AnalysisResultRepo) into a single unified repository and introduce the concept of projects to organize and segregate conversations, messages, engine sessions, and analysis results by project context.

## Requirements
- [ ] Create Project schema with fields (id, name, description, settings, created_at, updated_at)
- [ ] Add project_id foreign key to all existing entities (conversations, messages, engine_sessions, analysis_results)
- [ ] Create unified Repository module replacing all individual repository modules
- [ ] Implement project-scoped operations for all data entities
- [ ] Ensure data isolation between projects (no cross-project queries)
- [ ] Migrate existing data to default project during transition
- [ ] Update Transaction module to work with unified repository
- [ ] Update all service layer calls to use unified repository with project context
- [ ] Maintain existing semantic function naming (add, remove, change)
- [ ] Preserve all existing functionality and performance characteristics

### Technical Approach

#### Phase 1: Project Schema and Database Changes
1. Create Project Ecto schema with validations and changesets
2. Create migration to add projects table
3. Create migrations to add project_id to existing tables
4. Add foreign key constraints and indexes
5. Create data migration to assign existing records to default project

#### Phase 2: Unified Repository Implementation
1. Create new Repository module with project-scoped operations
2. Implement all CRUD operations with project context enforcement
3. Add project management operations (add, remove, change projects)
4. Implement cross-entity queries within project scope
5. Add project-aware caching and performance optimizations

#### Phase 3: Migration and Integration
1. Update Transaction module to use unified repository
2. Update all service layer code to use unified repository
3. Remove old repository modules
4. Update tests to use unified repository with project context
5. Verify all functionality preserved

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Data migration complexity | High | Create comprehensive data migration scripts with rollback capability |
| Performance degradation | Medium | Add proper indexes and optimize queries, benchmark before/after |
| Breaking existing functionality | High | Comprehensive test coverage and gradual migration approach |
| Project context propagation errors | Medium | Strict typing and validation of project context in all operations |
| Transaction complexity increase | Medium | Maintain existing transaction patterns but with unified repository |
